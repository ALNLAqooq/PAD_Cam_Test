# PAD 上 Qt 相机能力验证说明

更新时间：2026-03-10  
当前基线分支：`main`

---

## 1. 文档目的

这份文档用于记录：

- 当前 `main` 分支基于 Qt 的相机方案已经验证到什么程度
- 在 PAD 真实设备上，哪些能力可用，哪些能力不可用
- 对焦、闪光灯、手电筒这类能力为什么暂时不建议继续在 Qt 方案上深挖
- 如果后续要继续实现更强控制能力，应该往哪条技术路线走

这份文档的核心结论是：

> 对这台 PAD 的后置摄像头而言，Qt 方案已经能够实现高分辨率预览与拍照，但无法从 Qt 当前接口中获取点击对焦和闪光灯/手电筒能力。

---

## 2. 当前项目在 `main` 分支上的能力

当前 `main` 分支使用的是 Qt 相机方案，核心依赖为：

- `QCamera`
- `QCameraImageCapture`
- `QCameraViewfinder`
- `QVideoProbe`
- `QCameraFocus`
- `QCameraExposure`

当前已验证可用的能力如下：

### 2.1 可用能力

- 枚举摄像头设备
- 打开指定摄像头
- 实时预览
- 查看 Qt 返回的预览档位
- 查看 Qt 返回的拍照分辨率
- 切换预览档位
- 切换拍照分辨率
- 拍照并保存为图片

### 2.2 已验证成功的设备能力

在 PAD 上选中后置摄像头后，Qt 返回的设备信息如下：

- 当前摄像头：`Camera Sensor Rear`
- 最高预览档位：`3840 x 2160 | 30fps`
- 最高拍照分辨率：`3840 x 2160`

也就是说：

- 这台 PAD 的后置摄像头，在 Qt 路线上至少已经能稳定拿到 4K 级别的预览和拍照能力

这部分是当前项目最稳定、最可用的成果。

## 3. 本次专门验证的两类能力

为了进一步判断这台 PAD 是否适合继续在 Qt 路线上扩展更多相机控制，本次专门验证了两类能力：

### 3.1 对焦能力

检测内容包括：

- 对焦接口是否可用
- 是否支持自动对焦
- 是否支持连续对焦
- 是否支持中心对焦点
- 是否支持自定义对焦点
- 是否支持点击对焦

### 3.2 闪光灯 / 手电筒能力

检测内容包括：

- 曝光接口是否可用
- 闪光灯是否已就绪
- 是否支持：
  - `FlashOff`
  - `FlashOn`
  - `FlashAuto`
  - `FlashTorch`
  - `FlashVideoLight`

这里要特别区分两个概念：

- **Flash**
  - 更偏向拍照瞬间补光
- **Torch / VideoLight**
  - 更偏向常亮补光 / 手电筒

## 5. 这些日志说明了什么

### 5.1 对焦方面的结论

Qt 返回结果显示：

- `QCameraFocus` 在这台设备上不可用
- 自动对焦不可用
- 连续对焦不可用
- 中心对焦不可用
- 自定义对焦点不可用
- 点击对焦不可用

这说明：

> 当前 Qt 相机后端没有把这台 PAD 后置摄像头的对焦控制能力暴露出来。

这里的“不支持”应该理解为：

- **Qt 当前暴露的能力不支持**
- 不一定能直接推导为：
  - 硬件物理上绝对不支持
  - 系统底层绝对没有这项能力

换句话说：

- 不能简单下结论说“设备没有自动对焦”
- 更准确的结论是：
  - **当前 Qt 方案拿不到这项能力**

### 5.2 闪光灯方面的结论

Qt 返回结果显示：

- `QCameraExposure` 对象存在
- 说明“曝光接口”本身是可访问的

但是：

- `isFlashReady()` 不支持
- `FlashOff / FlashOn / FlashAuto / FlashTorch / FlashVideoLight` 全都不支持

这说明：

> Qt 能拿到这台相机的“曝光接口”，但拿不到它的“闪光灯 / 手电筒能力”。

同样地，这里的“不支持”也应理解为：

- **Qt 当前相机后端未暴露出闪光灯控制能力**
- 不一定能直接推出：
  - 这台 PAD 物理上没有灯
  - Windows 系统底层完全无法控制该灯

## 6. 当前应该如何理解“Qt 不支持”

这里非常重要。

### 6.1 不是说硬件一定没有

用户已明确说明：

- 这台 PAD 本身带有闪光灯

这意味着：

- 设备层面可能有真实的补光 / 闪光能力

同理，对焦也是一样：

- 即使 Qt 检测不到，也不代表底层绝对做不到

### 6.2 更可能是 Qt 路线没有暴露

更合理的技术判断是：

- 在 Windows 上，Qt 5 的当前相机后端没有把该设备的某些高级控制能力映射出来

这类能力包括：

- 点击对焦
- ROI 对焦
- 自动对焦控制
- 闪光灯模式
- 手电筒模式

因此：

> 当前的阻塞点并不是“功能想不到怎么做”，而是“Qt 这条路线拿不到真实可控能力”。

## 7. 现阶段的明确结论

针对这台 PAD 的后置摄像头，当前可以明确写下来的结论如下：

### 7.1 可以继续使用 Qt 方案做的事

- 高分辨率预览
- 高分辨率拍照
- 设备能力探测
- 档位切换
- 图像保存

### 7.2 不建议继续在 Qt 方案上深挖的事

- 点击对焦
- ROI 对焦
- 自动对焦控制
- 闪光灯控制
- 手电筒控制

原因不是这些功能不重要，而是：

- 当前设备上的实测结果已经表明，Qt 这条路拿不到这些能力

继续深挖的收益非常有限。

---

## 8. 为什么之前尝试了 Windows Native 路线

之前已经在一个单独分支中尝试过：

- `feat/windows-native-camera-focus`

该分支中主要完成了：

- 相机后端抽象
- `QtCameraBackend`
- `WindowsNativeCameraBackend`
- Windows 原生设备枚举
- 原生预览
- 原生档位枚举
- 原生基础拍照

但在继续推进“点击对焦 / ROI 对焦”时，出现了新的问题：

- 真正适合做“像微软相机那样的点击对焦”的路线，更接近：
  - `MediaCapture`
  - `WinRT`
  - `C++/WinRT`

而当前开发机环境缺少完整 Windows 10/11 SDK / WinRT 开发支持，因此该方向暂时暂停。

---

## 9. 如果后续还想做闪光灯 / 点击对焦，应该怎么做

如果后续还想继续做：

- 点击对焦
- ROI 对焦
- 闪光灯
- 手电筒

建议不要继续在当前 `main` 的 Qt 路线上深挖，而应该转向：

### 推荐路线

- Qt 负责界面
- Windows 原生相机栈负责设备控制

更具体地说：

- `MediaCapture / WinRT`

这条路线更接近微软自带相机应用实际使用的控制能力。

### 但当前阻塞点是

开发机当前缺少：

- Windows 10/11 SDK
- `C++/WinRT`
- 对应 WinRT 元数据与工具链

因此在环境补齐之前，这部分不建议继续投入时间。

---

## 10. 目前的建议

### 10.1 如果目标是“稳定可用”

建议当前就把项目定位为：

- 高分辨率 PAD 摄像头探测 / 预览 / 拍照工具

也就是说，把它作为：

- 设备能力查看工具
- 高分辨率相机验证工具
- 拍照保存工具

这是当前最稳妥、最有价值的使用方式。

### 10.2 如果目标是“高级相机控制”

如果未来仍要实现：

- 点击对焦
- 闪光灯
- 手电筒

建议后续条件允许时再继续：

1. 补齐 WinRT 开发环境
2. 恢复或重建 Windows Native 后端路线
3. 将控制能力迁移到更合适的原生相机栈

---

## 11. 一句话总结

> 在这台 PAD 的后置相机上，Qt 当前已经足够完成高分辨率预览和拍照，但无法从 Qt 接口中获取点击对焦和闪光灯/手电筒控制能力；如果未来还要实现这些高级功能，应考虑切换到 Windows 原生相机控制路线。
